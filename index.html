HRIS‑Driven Identity Lifecycle

How a mid‑size SaaS company wires its HR system (Paylocity) to its identity platform (Okta) so that hiring, transferring, and terminating an employee in HR automatically creates, adjusts, and removes their access everywhere else — with GCP hosting the glue in the middle. Written for engineers who know Entra ID and endpoint management but are new to Okta’s vocabulary: every identity term is defined the first time it appears.

Architecture Overview

Read the diagram left to right. Paylocity is the HRIS (Human Resources Information System — the HR database where employees are hired, transferred, and terminated) and it acts as the system of record (the one authoritative source that every other system copies from, never the other way around). When HR creates or changes a person there, Okta — the IdP (Identity Provider — the central login and identity platform; the Okta equivalent of Entra ID) — imports that person and builds a user profile from mapped attributes. Group rules (saved expressions that read profile attributes like department and automatically place users into groups — roughly Entra ID dynamic groups) put each user into the right groups; groups carry app assignments (which applications a member gets — like assigning users to an enterprise app in Entra); and each assigned app is then provisioned over SCIM (System for Cross-domain Identity Management — an open REST standard that lets an identity platform automatically create, update, and deactivate user accounts in other apps; the plumbing that copies a new hire from HR into every downstream tool). Google Workspace, Atlassian, and Zscaler take that feed more or less directly; 1Password cannot, so a small SCIM Bridge container hosted on GCP translates in between. Jamf, the MDM (Mobile Device Management — remote management of laptops and phones, the Intune of the Apple world), handles the devices those people use, alongside the account pipeline rather than inside it. The chain to memorize: attributes → group rules → group membership → app assignment → SCIM provisioning.

Entra bridge: if you think in Microsoft terms, this whole page describes the Okta version of “HR-driven provisioning” — the same idea as Entra ID’s HR-sourced provisioning into Active Directory / Entra, with dynamic groups and enterprise-app provisioning downstream. Okta = Entra ID; Okta group rules = dynamic groups; Okta app assignment + SCIM = enterprise app assignment + automatic provisioning; Okta System Log = Entra audit/sign-in logs; Jamf = Intune for Macs.
P0

Phase 0 — Discovery & Current‑State Audit

Read‑only. Nothing is written to any system in this phase.

Purpose

Before automating anything, learn exactly what exists today. Automation amplifies whatever it finds — clean data becomes clean accounts at scale, and dirty data becomes dirty accounts at scale. Phase 0 is a survey pass with every tool in read-only mode: you inventory the HR side, the Okta side, and every downstream app, and you write down what you find. Nothing is created, changed, or deleted.

Inputs

  • Read-only (or reporting) access to Paylocity, Okta, and each downstream app’s admin console.
  • The contract or plan documentation for Paylocity — you specifically need to know the tier, because Paylocity gates SCIM behind its Enterprise tier, and SAML (Security Assertion Markup Language — the older, XML-based single sign-on protocol most enterprise apps speak) setup requires contacting Paylocity support; it is not a self-serve toggle.
  • A list of every SaaS app in scope, with owner names.

Steps

  1. Audit Paylocity data quality. Export the employee roster and check attribute cleanliness: does every active employee have an employee ID, a department, a title, a manager, a start date? Blank or inconsistent values here become provisioning failures later. Note Paylocity’s username length restrictions — they can break an email-as-username convention if your emails are long.
  2. Classify each downstream app as (a) supports SCIM natively via the OIN (Okta Integration Network — Okta’s catalog of pre-built app connectors, analogous to the Entra enterprise application gallery), (b) needs glue such as Okta Workflows (Okta’s no-code automation product — think Logic Apps or Power Automate for identity events) or custom middleware, or (c) manual-only.
  3. Map the existing Okta group structure. Which groups exist, which are hand-maintained, which drive app assignments, and which are stale.
  4. Count existing users and determine how each was created — imported, API-created, or hand-typed by an admin years ago. This census is the input to Phase 1.
  5. Record every incomplete HR record. Paylocity provisioning can fail silently when HR-module records are incomplete — the user simply never syncs, and nobody is alerted. An unsynced account that quietly keeps stale access is an audit finding waiting to happen, so the gaps must be on paper before anything runs.

Outputs

  • An audit document: app inventory with SCIM capability per app, Okta group map, user census with creation origin.
  • A data-quality report: per-attribute completeness percentages from Paylocity, plus a named list of records that would fail to sync.

Done when

You can answer, without logging into anything: how many users exist, where each came from, which apps can take SCIM, which need glue, and which HR records are too incomplete to sync. You know exactly what you’re working with.

Risks & gotchas

  • Skipping the Paylocity tier check and discovering mid-project that SCIM isn’t in the contract.
  • Assuming “SSO is set up” means provisioning is set up. SSO (Single Sign-On — logging into many apps with one identity) only handles authentication; SCIM handles account lifecycle. They are separate integrations.
  • Trusting the HR data because “payroll works.” Payroll only needs pay-related fields; identity needs department, title, and manager, which payroll never exercises.

EXAMPLE — read-only census of Okta users by creation source (dummy tenant)

import os, requests

BASE = "https://dev-000000.okta.com/api/v1"
HDRS = {"Authorization": f"SSWS {os.environ['OKTA_API_TOKEN']}"}

counts = {}
url = f"{BASE}/users?limit=200"
while url:
    r = requests.get(url, headers=HDRS)
    r.raise_for_status()
    for u in r.json():
        src = u.get("credentials", {}).get("provider", {}).get("type", "unknown")
        counts[src] = counts.get(src, 0) + 1
    url = r.links.get("next", {}).get("url")   # follow pagination

for src, n in sorted(counts.items()):
    print(f"{src:12} {n}")   # e.g. OKTA 412 (hand-created), IMPORT 96

“Phase zero is a read-only survey: I inventory the HR data, the identity platform, and every downstream app before automating anything, because automation only amplifies whatever data quality you already have.”

P1

Phase 1 — Legacy Reconciliation

Binding years of hand-made accounts to the pipeline without breaking anyone.

Purpose

Most organizations don’t start from zero — they start with hundreds of accounts created by hand over years, with attributes that have drifted or were never filled in. Reconciliation (systematically matching records in two systems and resolving the differences, like reconciling a bank statement) binds those existing accounts to their HR records so the pipeline manages them from now on. The goal is captured in one framing: seamless to the end user, staged and deliberate for the operator. No employee should notice anything; the operator should see every single change before it happens.

Inputs

  • Phase 0’s user census and data-quality report.
  • An export of active employees from Paylocity with employee ID, email, department, title, and manager.
  • The target attribute schema — the same set of profile fields a brand-new hire would get in Phase 2.

Steps

  1. Match, don’t create. Key each Paylocity record to an existing Okta user on employee ID (preferred, because it never changes) or email (fallback). A matching pass that can’t find a stable key must stop, not create — otherwise the pipeline spawns a duplicate next to the real account, and in most SaaS targets a duplicate email either fails outright or forks the person’s identity in two.
  2. Attribute backfill (filling in missing or stale profile fields on existing accounts from the authoritative source): compute, for every matched user, the delta between their current Okta profile and what Paylocity says it should be — so legacy profiles end up on the exact same schema new hires get.
  3. Read-only diff first. The script’s default mode reports every change it would make — total counts and per-user before/after values — and writes nothing. A human reads that report. Only then is the same script re-run with an explicit apply flag.
  4. Exception queue: accounts with no clean HR source — contractors, service accounts, shared mailboxes — are surfaced into a list and resolved deliberately: by hand, or by writing an explicit exception rule. They are never silently skipped and never guessed at.

Outputs

  • A diff report (kept in Git) showing every proposed change before it was applied.
  • Legacy Okta profiles populated to the same schema as pipeline-born users, each linked to its HR record.
  • A worked-down exception queue with a documented decision per account.

Done when

A legacy user and a new hire are indistinguishable in Okta — same attribute schema, same HR linkage — and the exception queue is empty or every entry has a documented owner and decision.

Risks & gotchas

  • Duplicate creation is the classic failure: matching on a sloppy key (display name, or an email that was re-issued to a new hire) creates a second account and splits access history across two identities.
  • Overwriting better data with worse: sometimes Okta has the correct title and HR is stale. The diff report exists precisely so a human catches this before apply.
  • Service accounts swept into the people pipeline. A shared mailbox “deactivated because it has no HR record” is a self-inflicted outage. That’s what the exception queue is for.

EXAMPLE — reconciliation diff, dry-run by default, idempotent apply

import os, sys, requests

BASE = "https://dev-000000.okta.com/api/v1"
HDRS = {"Authorization": f"SSWS {os.environ['OKTA_API_TOKEN']}"}
APPLY = "--apply" in sys.argv          # default is a read-only dry run

def reconcile(hr_row, okta_user):
    """hr_row: from Paylocity export; okta_user: matched on employeeNumber."""
    profile = okta_user["profile"]
    want = {"department": hr_row["department"], "title": hr_row["title"]}
    delta = {k: v for k, v in want.items() if profile.get(k) != v}
    if not delta:
        print(f"[skip]  {profile['login']} already correct")
        return
    for k, v in delta.items():
        print(f"[plan]  {profile['login']}: {k}: {profile.get(k)!r} -> {v!r}")
    if APPLY:
        requests.post(f"{BASE}/users/{okta_user['id']}",
                      headers=HDRS, json={"profile": delta}).raise_for_status()
        print(f"[done]  {profile['login']}")

# alice@example.com matched on employeeNumber E10042 -> backfill runs
# svc-scanner@example.com has no HR record -> goes to the exception queue
Entra bridge: this is the same discipline as a staged Entra Connect rollout — you review the export of pending changes before enabling sync, and you use soft-match on a stable anchor attribute rather than letting the sync engine create duplicates.

“Legacy reconciliation means binding years of hand-made accounts to the HR feed: match on employee ID so we never create duplicates, backfill attributes from HR, and run everything as a read-only diff a human reviews before a single write happens. When it’s done, a five-year-old account looks exactly like tomorrow’s new hire.”

P2

Phase 2 — Joiner Path

New hire in Paylocity → working accounts everywhere, untouched by hand.

Purpose

Automate day one. When HR completes a hire in Paylocity, the pipeline imports the person into Okta, builds their profile from mapped attributes, places them into groups by rule, assigns apps by group, and provisions accounts downstream over SCIM — no ticket, no checklist, no admin typing a name twice.

Inputs

  • The Paylocity → Okta integration from the OIN, enabled and scheduled (this is “Paylocity as a source” — the same role HR-driven provisioning plays in Entra).
  • A reviewed attribute-mapping table (below) — the contract between HR’s schema and identity’s schema.
  • Group rules and group-to-app assignments, created in advance and tested against seeded accounts.

Steps

  1. HR completes the hire in Paylocity with all required fields.
  2. Okta’s scheduled import picks up the new record and creates the user via attribute mapping (per-field rules that translate a source system’s fields into the identity platform’s fields — Entra calls these attribute mappings too, in provisioning configuration).
  3. Group rules evaluate the new profile — e.g. user.department == "Engineering" — and add the user to groups automatically. Nobody hand-picks groups.
  4. Group membership triggers app assignments; each app’s provisioning integration then creates the downstream account over SCIM (or the app’s API — see the app notes below).
  5. An Okta Workflows flow resolves the manager attribute — see the deep-dive below, because Paylocity does not hand you a usable manager value.

Outputs

  • An activated Okta user with a complete, HR-sourced profile.
  • Accounts in Google Workspace, Atlassian, Zscaler, and 1Password, created by the pipeline.
  • System Log entries documenting every step — the audit trail is a by-product, not extra work.

The attribute-mapping table — the load-bearing artifact

Paylocity’s schema is HR-centric: cost centers, pay groups, EEO codes — fields built for payroll and compliance reporting. Okta’s schema is identity-centric: login, department, title, manager — fields that drive access. The mapping table is where one worldview is translated into the other, and it deserves the most review of anything in this phase, because bad mapping = wrong access. If Paylocity’s cost-center code is mapped into department, the group rules downstream will happily grant the wrong apps to the wrong people, at scale, with perfect reliability.

Deep dive: the manager-attribute problem (and the Workflows fix)

Paylocity passes the manager as an external ID unique to Paylocity — an opaque internal number, not an email or a reference Okta can resolve to another user. Left alone, the manager field is useless. The fix pattern is an Okta Workflows flow that: (1) lists Paylocity-assigned users, (2) matches each user’s external manager ID against the other users’ own external IDs, and (3) writes the resolved manager (their Okta login) back onto the user’s profile.

Why bother? Manager data drives access-approval routing (who approves this person’s request for an app) and access-certification campaigns (periodic “does this person still need this?” reviews that auditors require — Entra calls these access reviews). A blank manager field silently breaks both.

Done when

A test hire entered only in Paylocity results — with zero human touches — in an Okta account with correct attributes, correct group memberships, a resolved manager, and working downstream accounts, all within one import cycle.

Risks & gotchas

  • Silent non-provisioning: an incomplete Paylocity HR record can simply never sync, with no error raised. Pair the pipeline with a scheduled report comparing HR headcount to Okta headcount.
  • Mapping drift: HR renames a department, and every group rule keyed on the old string stops matching. Treat department values as a controlled vocabulary shared between HR and IT.
  • Rules that only add: a group rule places users in; make sure the same rule set is what keeps them there, so a later attribute change moves them (Phase 3) instead of accumulating.

EXAMPLE — group rule (concept) and what SCIM sends downstream

# Group rule (Okta expression language), maintained in Git:
#   IF   user.department == "Engineering"
#   THEN assign to groups: eng-all, vpn-eng, atlassian-users
# ≈ an Entra dynamic group with rule (user.department -eq "Engineering")

# When group membership assigns an app, Okta sends the SCIM create:
scim_create = {
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
    "userName": "alice@example.com",
    "name": {"givenName": "Alice", "familyName": "Nguyen"},
    "emails": [{"value": "alice@example.com", "primary": True}],
    "active": True,
    "externalId": "E10042",     # the stable key — duplicate prevention
}
# POST {app}/scim/v2/Users -> 201 Created (or 409 if the key already exists:
# match, don't create — bind to the existing account instead)

Downstream app notes

  • Google Workspace is a downstream app here — Okta is the master. Okta’s connector drives Google’s Admin SDK Directory API under the hood, even though the lifecycle model in Okta looks like SCIM. Don’t blur Google Workspace with GCP: different products sharing an identity backbone. Okta → Cloud Identity → GCP IAM is a separate federation chain for cloud-console access.
  • Atlassian requires Atlassian Guard (their paid identity add-on, formerly Atlassian Access) before SAML or SCIM works at all — budget for it in Phase 0.
  • Zscaler consumes Okta groups via SCIM, so network access policy binds to group membership rather than IP ranges — a department change re-scopes network access automatically.
  • 1Password can’t accept SCIM directly (its end-to-end encryption model prevents the vendor’s cloud from processing provisioning data), so a self-hosted 1Password SCIM Bridge container runs on GCP — Cloud Run or GKE — between Okta and 1Password. This is GCP’s whole role in the pipeline: middleware hosting, not a provisioning target.
  • Jamf Connect binds macOS login to Okta credentials, so local laptop accounts don’t drift from the identity platform (same goal as Entra hybrid join keeping Windows logins tied to the directory).

“The joiner path means HR enters a hire once, and the pipeline does the rest: import into Okta, attributes drive group rules, groups drive app assignments, and SCIM creates the downstream accounts. The attribute-mapping table is the artifact I’d review hardest — bad mapping is wrong access at scale.”

P3

Phase 3 — Mover Path

Transfers and role changes — where most pipelines silently fail.

Purpose

Handle the middle of the lifecycle: transfers, promotions, department changes. The rule is unforgiving: a department change must grant new access AND revoke old access. Half of that is easy; the other half is where pipelines rot.

Say it explicitly: movers are where most identity pipelines silently fail. Half-built pipelines handle hire and termination only. When someone transfers, the new team requests their new access by ticket, nobody thinks to remove the old, and the person accumulates entitlements with every move. Orphaned access from an old role is a textbook audit finding — the ten-year employee who has moved four times and can still touch the finance share, the prod console, and the old team’s admin panel.

Inputs

  • The department/title change recorded in Paylocity (the mover “event” is just an attribute change on the next import — there is no separate transfer API).
  • Group rules written as complete descriptions of who belongs, so a rule that adds Engineering members also, by ceasing to match, removes ex-Engineering members.
  • SCIM provisioning configured for update and deprovision, not just create.

Steps

  1. HR updates the employee’s department in Paylocity.
  2. The next Okta import updates the user’s profile attribute.
  3. Group rules re-evaluate: the user enters the new department’s groups and leaves the old ones — the leave is automatic only because membership is rule-derived rather than hand-granted.
  4. App assignments follow group membership: new apps are provisioned via SCIM; apps granted only through the old groups are deprovisioned.
  5. Apps shared by both roles receive a SCIM update (new department, new groups) rather than a churn of delete-and-recreate.
  6. Zscaler policy re-scopes automatically, since its policy binds to the Okta groups that just changed.

Outputs

  • Access that matches the new role — additions and removals — within one import cycle.
  • System Log evidence of both halves, which is exactly what an access review wants to see.

Done when

A test transfer (change one seeded user’s department in Paylocity) produces both grants and revocations with no human involvement — and you can point to the revocations in the log, not just the grants.

Risks & gotchas

  • Hand-granted extras survive moves. Rules only remove what rules granted. Access granted directly to a user outside the group model is invisible to the mover path — which is the argument for allowing direct assignment only as a logged exception.
  • Data loss on revoke: leaving a team can remove the app that holds their old files. Decide the retention pattern per app (e.g., Google Workspace content transfer) before the first real mover, not during.
  • Timing gaps: HR often keys transfers to a future effective date; make sure the import honors it rather than moving access weeks early.

EXAMPLE — verifying a mover produced revocations, not just grants (dry-run pattern)

import os, sys, requests

BASE = "https://dev-000000.okta.com/api/v1"
HDRS = {"Authorization": f"SSWS {os.environ['OKTA_API_TOKEN']}"}
APPLY = "--apply" in sys.argv

def mover_check(user_id, old_dept_groups, new_dept_groups):
    r = requests.get(f"{BASE}/users/{user_id}/groups", headers=HDRS)
    r.raise_for_status()
    current = {g["profile"]["name"] for g in r.json()}
    to_gain  = new_dept_groups - current
    to_lose  = old_dept_groups & current
    for g in sorted(to_gain):
        print(f"[plan]  GRANT  {g}")
    for g in sorted(to_lose):
        print(f"[plan]  REVOKE {g}   # the half everyone forgets")
    if not APPLY:
        print("[dry-run] no changes made; re-run with --apply")

mover_check("00u000000000000000a0",
            old_dept_groups={"eng-all", "vpn-eng"},
            new_dept_groups={"sales-all", "crm-users"})
Entra bridge: identical logic to dynamic-group membership changing when a user attribute changes — the difference is discipline, not technology. If access is granted only via rule-derived groups, moves are symmetric. Every direct assignment is a future audit finding.

“Movers are the test of whether a lifecycle pipeline is real: a department change has to revoke the old access, not just grant the new. Most half-built pipelines only handle hire and fire, and the leftover access from old roles is exactly what auditors find.”

P4

Phase 4 — Leaver Path

One action in HR → complete access removal in minutes, with evidence.

Purpose

Make termination a single action with total, provable reach: HR terminates the person in Paylocity, and the pipeline deactivates the identity, cuts access to every downstream app, and locks the hardware — in minutes, with timestamps.

Inputs

  • The termination recorded in Paylocity (with attention to the effective date and, for urgent departures, an out-of-band fast path — see gotchas).
  • Okta provisioning set to deactivate on source removal; SCIM deprovisioning enabled per app.
  • Jamf API access for device actions, keyed by the same user identity.

Steps

  1. HR terminates the employee in Paylocity.
  2. Okta’s import sees the person leave the source and deactivates the Okta user: sessions are killed, sign-in is blocked, and every SSO-connected app is unreachable at the front door immediately.
  3. Deprovisioning (the automated removal of access — the reverse of provisioning) fans out over SCIM: each downstream app receives an update setting active: false. Note carefully: this suspends the account rather than sending a DELETE. The object survives, which preserves audit evidence (who had access, when, what did they own) and makes rehires a reactivation instead of a rebuild.
  4. Device action via MDM: a Jamf lock (or wipe, per policy) hits the person’s laptop, closing the gap SSO can’t reach — cached local sessions and offline data.
  5. The run is verified against the Okta System Log (Okta’s unified event stream — the equivalent of Entra’s audit + sign-in logs), and those events flow to the SIEM (Security Information and Event Management — the central log platform, e.g. Splunk-class tooling) as durable audit evidence.

Outputs

  • A deactivated (not deleted) identity and suspended downstream accounts.
  • A locked or wiped device.
  • A timestamped trail from HR action to every downstream effect.

Done when

A test termination of a seeded account results, within minutes and with no human touches, in: no Okta sign-in, suspended accounts in every downstream app, a locked test device, and System Log entries for each step. One action in the HR system, complete access removal in minutes, with a timestamped audit trail.

Risks & gotchas

  • Import lag vs. urgent terminations: a scheduled HR import may run hourly or daily. For for-cause departures, have a documented fast path (suspend directly in Okta first; let the HR event reconcile behind it).
  • The apps outside the pipeline are the leaver path’s blind spot — anything classified “manual-only” in Phase 0 needs a human checklist triggered by the same event.
  • Deletion habits from on-prem: resist the urge to delete-after-30-days by reflex. Suspended objects are your evidence in an investigation and your shortcut in a rehire; deletion is a separate, deliberate retention decision.

EXAMPLE — what SCIM deprovisioning actually sends (suspend, not delete)

import os, sys, requests

APPLY = "--apply" in sys.argv
app_base = "https://scim.example.com/scim/v2"   # a downstream app's SCIM endpoint
HDRS = {"Authorization": f"Bearer {os.environ['SCIM_BEARER_TOKEN']}"}

def deprovision(scim_user_id, login):
    patch = {
        "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
        "Operations": [{"op": "replace", "path": "active", "value": False}],
    }   # active:false = suspend. NOT a DELETE — the object stays for audit/rehire.
    if not APPLY:
        print(f"[dry-run] would suspend {login} ({scim_user_id})")
        return
    r = requests.patch(f"{app_base}/Users/{scim_user_id}", headers=HDRS, json=patch)
    r.raise_for_status()
    print(f"[done] suspended {login}")

deprovision("2819c223-7f76-453a-919d-413861904646", "bob@example.com")

“The leaver path is the security payoff: HR terminates someone once, and within minutes Okta kills their sessions, SCIM suspends every downstream account — suspend, not delete, so the audit trail survives — and the MDM locks the laptop. One action, complete removal, timestamped evidence.”

SFT

Safety Model

Cross-cutting rules that apply to every phase and every script.

Identity automation is a lever that moves hundreds of accounts at once, so every write path in this pipeline obeys the same six rules. None of them are optional, and none are phase-specific.

1. Dry-run by default

Every script prints its plan and changes nothing unless explicitly told otherwise: --dry-run is the default behavior, --apply is the explicit opt-in. Every write-path example on this page shows the pattern. A tool whose default is “do it” will eventually be run by accident; a tool whose default is “show me” cannot be.

2. Blast-radius caps

Blast radius (the number of users or systems a single run can affect — the damage limit if something is wrong) is capped in code: if a run would touch more than N users, it halts and requires human confirmation. A reconciliation script that suddenly plans 400 changes when yesterday it planned 12 isn’t more productive — it’s telling you an upstream assumption broke.

MAX_TOUCHED = 25    # blast-radius cap

planned = build_plan(hr_export, okta_users)   # list of intended changes
if len(planned) > MAX_TOUCHED and not "--confirm-large-run" in sys.argv:
    print(f"[halt] plan touches {len(planned)} users (cap {MAX_TOUCHED}).")
    print("[halt] review the plan, then re-run with --confirm-large-run")
    sys.exit(1)

3. Pilot small

Every phase is proven on one department — or a set of seeded test accounts (deliberately created fake users like alice@example.com that exist to exercise the pipeline) — before going org-wide. The pilot population is where mapping mistakes and rule typos surface cheaply.

4. Deactivate, never delete

Deactivation is reversible and preserves evidence; deletion is neither. The pipeline suspends (active: false), and deletion happens later, if ever, as a separate deliberate retention decision — never as an automated side effect.

5. Observability

The Okta System Log is open during every applied run — you watch the events land as the script reports them, and mismatches stop the run. The same events feed the SIEM, which turns operational logging into audit evidence with zero extra effort. (Entra bridge: this is your habit of watching the audit log during a risky change, made mandatory.)

6. Everything in Git

Scripts, mapping tables, group-rule definitions, and dry-run reports all live in a repository: diffable (you can see exactly what changed between runs), reviewable (a second person approves before apply), and revertible (yesterday’s known-good config is one checkout away). Configuration that lives only in an admin console has none of those properties.

“Every write in this pipeline is dry-run by default, capped in blast radius, piloted small, and logged to the System Log and SIEM — and it suspends rather than deletes, with every script and mapping table in Git so changes are reviewed and revertible.”

How many users or systems one run can affect — capped in code so a bad run halts instead of scaling.
Deprovisioning ≈ Entra: account removal on deprovision
The automated removal of access when someone leaves or changes role — the reverse of provisioning; here it suspends rather than deletes.
Dry run
Running a script in report-only mode: it prints every change it would make and makes none. The default mode of every write path here.
Group rule ≈ Entra: dynamic group
A saved expression in Okta that reads profile attributes (like department) and places users into groups automatically.
HRIS
Human Resources Information System — the HR database (here, Paylocity) where people are hired, transferred, and terminated.
IAM
Identity and Access Management — the whole discipline of controlling who someone is and what they can reach.
Idempotent
Safe to run twice: an idempotent script skips what already exists ([skip]) instead of duplicating or erroring, so reruns are harmless.
IdP ≈ Entra ID itself
Identity Provider — the central platform that owns logins and vouches for users to other apps. Here, Okta.
JIT provisioning
Just-In-Time provisioning — creating an app account at the moment of first login (via SAML/OIDC) instead of ahead of time via SCIM. Convenient, but it can’t deactivate anyone — which is why this pipeline prefers SCIM.
MDM ≈ Intune
Mobile Device Management — remote management of laptops and phones (here, Jamf for Apple devices), including lock and wipe.
OAuth 2.0
The authorization framework that lets one service act on another’s API with scoped tokens instead of passwords; OIDC and most SCIM auth ride on it.
OIDC ≈ Entra: OpenID Connect apps
OpenID Connect — the modern, JSON/REST login protocol built on OAuth 2.0; the newer alternative to SAML for SSO.
RBAC
Role-Based Access Control — granting access by role/group membership instead of per person. This pipeline’s chain: attributes → group rules → groups → app assignments → SCIM.
Reconciliation
Systematically matching records across two systems (HR ↔ identity) and resolving differences — matching, never blind creation.
SAML ≈ Entra: SAML enterprise apps
Security Assertion Markup Language — the older, XML-based SSO protocol most enterprise apps support.
SCIM ≈ Entra: automatic provisioning
System for Cross-domain Identity Management — the open REST standard for automatically creating, updating, and deactivating user accounts in other apps.
SCIM bridge
A small translation service between an identity platform and an app that can’t accept SCIM directly — here, 1Password’s self-hosted bridge container running on GCP.
SSO ≈ Entra: seamless/enterprise SSO
Single Sign-On — logging into many apps with one identity. Handles authentication only; account lifecycle is SCIM’s job.
System of record
The one authoritative source for a piece of data — here Paylocity for people. Everything else copies from it, never into it.
